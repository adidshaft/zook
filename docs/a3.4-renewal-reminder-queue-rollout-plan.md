# A3.4 Renewal Reminder Queue Rollout Plan

Last updated: 18 June 2026

## Goal

Move renewal reminder creation and notification fan-out out of the cron request path and into the
durable queue proposed by `A3.1`.

This completes the remaining queue portion of Product Plan Part A item `A3.4`. The already-landed
code improves the current inline cron by batching lookups; this rollout plan covers the next step
without changing runtime behavior before the queue exists.

## Current state

`POST /api/cron/renewal-reminders` currently does all work inline:

- scans member subscriptions for 7, 3, and 1 day expiry windows
- preloads existing `SubscriptionReminder` rows for those subscriptions
- creates new `SubscriptionReminder` rows
- calls `createDirectNotification()` for each member reminder
- scans SaaS trials for 7, 3, 1, and 0 day windows
- preloads owners for trial organizations
- creates trial reminder rows and owner notifications

The batching work avoids obvious N+1 reads, but the cron request can still become slow as tenant
volume grows and notification creation still chains into push delivery while `A3.3` is not queued.

## Dependency on A3.1

`A3.4` should use the durable queue proposed in:

- `docs/a3.1-durable-queue-rollout-plan.md`

Required queue kind:

```ts
{
  queue: "reminders",
  kind: "renewal_reminder_batch",
  dedupeKey: `renewal_reminder_batch:${dateKey}`,
}
```

Do not replace the inline cron until the `BackgroundJob` table, worker claim path, and worker
endpoint/script are available in staging.

## Desired cron behavior

Current response after inline work:

```ts
return ok({
  processed: true,
  remindersCreated: totalCreated,
  remindersSkipped: totalSkipped,
  notificationsSent: totalNotified,
});
```

Target response after queue rollout:

```ts
return ok({
  processed: true,
  queued: true,
  reminderJob: {
    id: job.id,
    status: job.status,
    dateKey,
  },
});
```

If API consumers still expect counts, keep the legacy count fields present as `null` during the
transition:

```ts
remindersCreated: null,
remindersSkipped: null,
notificationsSent: null,
```

This is an API contract change for cron/operator observability, so roll it out behind a feature flag.

## Job payload

Use a small payload and let the worker calculate windows at execution time:

```ts
{
  dateKey: string;
  requestedAt: string;
  windows: {
    memberSubscriptionDays: [7, 3, 1];
    trialDays: [7, 3, 1, 0];
  };
}
```

Use an India business-date `dateKey` if ops wants reminders aligned to the primary market day.
Otherwise use UTC and document that choice in cron scheduling.

## Producer changes

In `POST /api/cron/renewal-reminders`:

1. Keep the existing `CRON_SECRET` authorization check.
2. Derive a `dateKey` from the cron execution date.
3. Enqueue `renewal_reminder_batch` with dedupe key `renewal_reminder_batch:${dateKey}`.
4. Return immediately with job state.
5. Keep the current inline code path behind `RENEWAL_REMINDERS_QUEUE_ENABLED=false`.

Recommended flag:

```text
RENEWAL_REMINDERS_QUEUE_ENABLED=false
```

## Worker behavior

The worker should preserve current business behavior:

1. Process member subscription windows for 7, 3, and 1 day expiry.
2. Process SaaS trial windows for 7, 3, 1, and 0 days.
3. Keep the existing `take: 500` batch bound per window.
4. Preload plans, existing reminders, and owners using the already-landed batched queries.
5. Create `SubscriptionReminder` rows before notifications.
6. Use `createDirectNotification()` for now; after `A3.3`, notification creation should enqueue push
   instead of sending push inline.
7. Record `remindersCreated`, `remindersSkipped`, and `notificationsSent` in the job result.
8. Treat individual notification failures as best-effort skips, matching current cron behavior.
9. Retry job-level database/provider failures with exponential backoff.
10. Mark the job `DEAD` after `maxAttempts`.

Recommended job result shape:

```ts
{
  dateKey: string;
  memberWindows: [7, 3, 1];
  trialWindows: [7, 3, 1, 0];
  remindersCreated: number;
  remindersSkipped: number;
  notificationsSent: number;
  notificationFailures: number;
}
```

## Idempotency rules

Reminder creation must remain safe to replay.

Recommended dedupe markers:

- job dedupe: `renewal_reminder_batch:${dateKey}`
- member reminder uniqueness: existing check by `subscriptionId`, `kind`, `status`, and due window
- trial reminder uniqueness: existing check by `orgId`, `kind`, `status`, and
  `metadata.daysRemaining`

Before adding database uniqueness constraints for reminders, run a staging duplicate audit. Existing
data may already contain duplicate reminders if the inline cron was retried after partial work.

## Staging rollout steps

1. Apply and verify the A3.1 `BackgroundJob` migration on staging.
2. Deploy queue helpers and the reminders worker.
3. Add producer code behind `RENEWAL_REMINDERS_QUEUE_ENABLED=false`.
4. Run staging with the flag disabled and confirm current inline response counts are unchanged.
5. Enable the flag on staging.
6. Trigger `POST /api/cron/renewal-reminders`.
7. Confirm the HTTP response returns immediately with a job id.
8. Run the worker and confirm reminder rows/notifications match the inline baseline.
9. Replay the cron request for the same `dateKey` and confirm no duplicate reminders.
10. Force a worker failure after reminder creation and confirm retry does not duplicate rows.

Print a single section when needed:

```bash
pnpm db:a3.4:sql reminder-counts-by-kind
pnpm db:a3.4:sql reminder-notification-sample
pnpm db:a3.4:sql unfinished-reminder-jobs-before-rollback
```

Queued reminder jobs:

```sql
SELECT "id", "status", "queue", "kind", "dedupeKey", "attempts", "result", "lastError", "createdAt"
FROM "BackgroundJob"
WHERE "queue" = 'reminders'
  AND "kind" = 'renewal_reminder_batch'
ORDER BY "createdAt" DESC
LIMIT 20;
```

Duplicate member reminders:

```sql
SELECT "subscriptionId", "kind", date_trunc('day', "dueAt") AS due_day, COUNT(*) AS reminder_count
FROM "SubscriptionReminder"
WHERE "subscriptionId" IS NOT NULL
  AND "kind" = 'SUBSCRIPTION_EXPIRING'
  AND "status" IN ('PENDING', 'SENT')
GROUP BY "subscriptionId", "kind", due_day
HAVING COUNT(*) > 1
ORDER BY reminder_count DESC;
```

Duplicate SaaS trial reminders:

```sql
SELECT
  "orgId",
  "kind",
  "metadata"->>'daysRemaining' AS days_remaining,
  COUNT(*) AS reminder_count
FROM "SubscriptionReminder"
WHERE "kind" = 'SAAS_TRIAL_END'
  AND "status" IN ('PENDING', 'SENT')
GROUP BY "orgId", "kind", days_remaining
HAVING COUNT(*) > 1
ORDER BY reminder_count DESC;
```

## Rollback

If queued reminders fail in staging:

1. Set `RENEWAL_REMINDERS_QUEUE_ENABLED=false`.
2. Stop the reminders worker.
3. Keep queued jobs and reminder rows for inspection.
4. Re-enable the existing inline cron path.

If production rollout fails after jobs are enqueued, do not delete jobs until operators decide
whether to retry or intentionally abandon that date's reminder batch.

## Production checklist

- A3.1 queue migration and worker are deployed
- reminder worker can run independently from web request handling
- feature flag is available and defaults off
- staging confirms queued output matches inline baseline counts
- duplicate audits are clean or explicitly accepted
- replaying the same date does not duplicate reminders
- web gates pass: `pnpm --filter @zook/web typecheck`, `lint`, and `test`
- cron/operator copy is updated to say queued vs completed where counts are not immediate

## Out of scope in this proposal

- implementing the durable queue table (`A3.1`)
- moving platform broadcast fan-out (`A3.2`)
- moving push delivery (`A3.3`)
- adding reminder uniqueness constraints before staging duplicate audits
