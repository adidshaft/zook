# A1.7 Visit Deduction Uniqueness Plan

Last updated: 16 June 2026

## Goal

Enforce the product-plan invariant for visit deductions:

- one `MembershipUsage` row per `(subscriptionId, attendanceId)`

This is the Part A P0 item `A1.7` from `zook-product-improvement-plan.md`.

## Why this is still needed

Current application behavior already treats `MembershipUsage` as one row per attendance event:

- `applyAttendanceUsage()` first checks `membershipUsage.findFirst({ orgId, attendanceId })`
- if a usage row already exists for the attendance record, it returns without consuming again
- otherwise it decrements `remainingVisits` and inserts a `MembershipUsage` row

That is good intent, but the database does not enforce the invariant. The current schema has:

```prisma
model MembershipUsage {
  id             String   @id @default(cuid())
  orgId          String
  subscriptionId String
  attendanceId   String?
  usedVisits     Int      @default(1)
  usageDate      DateTime @default(now())
  metadata       Json?
  createdAt      DateTime @default(now())

  @@index([orgId, subscriptionId])
}
```

So concurrent retries can still create duplicate usage rows or double-consume visits unless application logic wins every race.

## Current schema target

Desired invariant from the plan:

```sql
CREATE UNIQUE INDEX membership_usage_unique_subscription_attendance_idx
ON "MembershipUsage" ("subscriptionId", "attendanceId")
WHERE "attendanceId" IS NOT NULL;
```

The `WHERE "attendanceId" IS NOT NULL` clause matters because the current column is nullable.

## Why partial uniqueness is the safest fit

The plan calls for uniqueness on `(subscriptionId, attendanceId)`, but `attendanceId` is nullable today.

If we create a plain composite unique index without filtering nulls, Postgres still allows multiple null combinations, but the intent is clearer and safer when we explicitly scope the constraint to real attendance-linked usage rows.

That matches the current write path, which only inserts usage rows from attendance consumption with a real `attendanceId`.

## Preflight duplicate audit

Run this first on a staging clone or disposable copy of production:

```sql
SELECT
  "subscriptionId",
  "attendanceId",
  COUNT(*) AS usage_count,
  ARRAY_AGG("id" ORDER BY "createdAt" ASC) AS usage_ids
FROM "MembershipUsage"
WHERE "attendanceId" IS NOT NULL
GROUP BY "subscriptionId", "attendanceId"
HAVING COUNT(*) > 1
ORDER BY usage_count DESC, "subscriptionId", "attendanceId";
```

If this returns any rows, do not apply the index yet.

Detail query per collision set:

```sql
SELECT
  "id",
  "orgId",
  "subscriptionId",
  "attendanceId",
  "usedVisits",
  "usageDate",
  "metadata",
  "createdAt"
FROM "MembershipUsage"
WHERE "subscriptionId" = $1
  AND "attendanceId" = $2
ORDER BY "createdAt" ASC;
```

Useful supporting query to compare against the source attendance row:

```sql
SELECT
  mu."id" AS usage_id,
  mu."subscriptionId",
  mu."attendanceId",
  mu."usedVisits",
  ar."orgId",
  ar."branchId",
  ar."userId",
  ar."dateKey",
  ar."status",
  ar."checkedInAt"
FROM "MembershipUsage" mu
LEFT JOIN "AttendanceRecord" ar
  ON ar."id" = mu."attendanceId"
WHERE mu."attendanceId" = $1
ORDER BY mu."createdAt" ASC;
```

## Suggested remediation rule on staging

Do not auto-delete duplicates without review.

For each duplicate set:

1. Keep the earliest credible row unless later rows clearly represent the correct deduction.
2. Reconcile `usedVisits` against the subscription plan and the member’s actual remaining visits.
3. If duplicate rows already caused an incorrect `remainingVisits`, document and repair the parent `MemberSubscription` as part of staging cleanup.

This is important because duplicate usage rows can mean the denormalized `remainingVisits` value is already wrong.

## Staging dry-run procedure

1. Restore or clone staging from the latest production snapshot.
2. Run the duplicate audit query above.
3. If duplicates exist, document the cleanup and any `remainingVisits` corrections.
4. Apply cleanup on staging only.
5. Re-run the duplicate audit until it returns zero rows.
6. Apply the partial unique index SQL on staging.
7. Re-run visit-pack smoke tests:
   - first approved attendance creates one usage row
   - replayed attendance processing does not create a second usage row
   - same-day second check-in does not double-consume when multi-entry is off
   - multi-entry gyms still behave according to current product rules

Index application SQL for staging:

```sql
CREATE UNIQUE INDEX membership_usage_unique_subscription_attendance_idx
ON "MembershipUsage" ("subscriptionId", "attendanceId")
WHERE "attendanceId" IS NOT NULL;
```

Validation query after index creation:

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'MembershipUsage'
  AND indexname = 'membership_usage_unique_subscription_attendance_idx';
```

## Prisma migration note

Like `A1.5`, this is best treated as a manual-SQL migration step because of the partial `WHERE "attendanceId" IS NOT NULL` clause.

Do not model this as a plain `@@unique([subscriptionId, attendanceId])` in `schema.prisma` unless the column is first made non-null and the rollout plan is intentionally changed.

## Rollback

If the index needs to be reversed:

```sql
DROP INDEX IF EXISTS membership_usage_unique_subscription_attendance_idx;
```

If recorded through Prisma, follow the repo runbook and resolve the named migration as rolled back only after the index is removed and the database state is confirmed.

## Production rollout checklist

- `pnpm release:preflight` passes with the target env
- fresh database backup/snapshot exists
- staging clone dry-run is documented
- duplicate usage audit returned zero rows on the rollout target immediately before applying the index
- any affected `remainingVisits` corrections were documented on staging before rollout
- visit-pack/attendance replay smoke checks passed on staging after index creation

## Out of scope in this proposal

- `A1.8` desk-payment amount validation
- `A1.1` FK / referential integrity rollout
- broader reconciliation of historical `remainingVisits` beyond duplicate-linked fixes
