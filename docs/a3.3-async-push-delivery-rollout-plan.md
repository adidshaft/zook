# A3.3 Async Push Delivery Rollout Plan

Last updated: 18 June 2026

## Goal

Move push-provider delivery out of request handlers and into a durable background worker.

This is Product Plan Part A item `A3.3`.

This document is intentionally a rollout proposal only. It does not change runtime behavior because
the durable queue proposed in `A3.1` has not been migrated/applied yet, and push delivery timing is
observable by users, platform admins, and QA flows. The first safe step is to make the target queue,
idempotency, and rollout contract explicit.

## Current problem

`createDirectNotification()` creates in-app notification rows and then immediately calls
`deliverPushForNotification()`.

`deliverPushForNotification()` currently performs all push work inline:

- reads notification preferences
- reads active push devices
- calls the selected push provider with `sendBatch()`
- writes `PushDelivery` rows
- invalidates devices when the provider returns invalid-token results

That makes any route that creates a notification depend on push-provider latency and failure modes.
Examples include plan publish, attendance events, payments, shop orders, platform broadcasts, and
membership lifecycle events.

## Dependency on A3.1

`A3.3` should use the durable queue proposed in:

- `docs/a3.1-durable-queue-rollout-plan.md`

Required queue kind:

```ts
{
  queue: "notifications",
  kind: "push_delivery",
  dedupeKey: `push_delivery:${notification.id}`,
}
```

Do not replace inline delivery until the `BackgroundJob` table, worker claim path, and worker
endpoint/script are available in staging.

## Desired notification creation behavior

Current behavior:

```ts
await deliverPushForNotification({
  orgId,
  notification,
  userIds,
});
```

Target behavior after the queue rollout:

```ts
await enqueueBackgroundJob({
  queue: "notifications",
  kind: "push_delivery",
  dedupeKey: `push_delivery:${notification.id}`,
  orgId,
  payload: {
    notificationId: notification.id,
    requestedUserIds: userIds,
    requestedAt: new Date().toISOString(),
  },
});
```

The route should return after the in-app notification and recipients are committed. Push delivery
status becomes eventual and inspectable through `PushDelivery` and job state.

## Job payload

Keep the payload small and re-read authoritative data in the worker:

```ts
{
  notificationId: string;
  requestedUserIds: string[];
  requestedAt: string;
}
```

The worker should load the notification, recipients, preferences, and active devices at execution
time. This lets preference/device changes made after notification creation apply before push sends.

## Worker behavior

The worker should:

1. Load `Notification` by `notificationId`.
2. Exit successfully without provider calls if the notification no longer exists or `pushEnabled`
   is false.
3. Resolve intended users from `requestedUserIds`, then intersect with existing
   `NotificationRecipient` rows for the notification.
4. Apply `UserNotificationPreference` exactly as `deliverPushForNotification()` does today.
5. Load active, non-revoked `PushDevice` rows for eligible users.
6. Create initial `PushDelivery` rows with `QUEUED` status before calling the provider.
7. Send provider batches in bounded chunks.
8. Update each delivery row to `SENT` or `FAILED`.
9. Invalidate devices when provider results are `invalid_token`.
10. Retry transient provider errors with exponential backoff.
11. Mark the job `DEAD` after `maxAttempts` and leave failed `PushDelivery` rows inspectable.

Recommended job result shape:

```ts
{
  notificationId: string;
  eligibleUsers: number;
  devices: number;
  sent: number;
  failed: number;
  invalidatedDevices: number;
  provider: string;
}
```

## Idempotency rules

Push delivery must be safe to retry.

Recommended dedupe markers:

- job dedupe: `push_delivery:${notification.id}`
- delivery uniqueness: one active `PushDelivery` per `notificationId + deviceId + attempt group`
- worker skip rule: if a delivery for the notification/device is already `SENT`, do not resend
- worker retry rule: failed provider attempts may create/update a new attempt group, but must not
  produce duplicate successful sends for the same notification/device pair

Because the existing `PushDelivery` model does not include an attempt-group field or uniqueness
constraint, add those only after a staging data audit. Until then, the worker can use
`findFirst({ notificationId, deviceId, status: "SENT" })` as the minimum retry guard.

## API and UI contract

No public API response should promise that push was already delivered once the queue is enabled.

Preferred wording:

- in-app notification created
- push delivery queued
- delivery status available in notification/job diagnostics

Do not expose provider-specific delivery ids to gym staff. Keep provider detail in platform/admin
diagnostics and logs.

## Staging rollout steps

1. Apply and verify the A3.1 `BackgroundJob` migration on staging.
2. Deploy queue helpers and the notifications worker.
3. Add producer code behind a feature flag:
   - `PUSH_DELIVERY_QUEUE_ENABLED=false` by default
4. Run with the flag disabled and confirm current inline behavior is unchanged.
5. Enable the flag on staging.
6. Trigger representative notification sources:
   - plan publish
   - attendance approval/rejection
   - shop pickup update
   - membership payment event
   - platform broadcast after `A3.2`
7. Confirm route responses are not delayed by provider calls.
8. Confirm a worker run writes expected `PushDelivery` statuses.
9. Force provider failure and confirm retry/dead-letter behavior.
10. Replay the same job and confirm no duplicate successful sends.

## Validation queries

Queued push jobs:

```sql
SELECT "id", "status", "queue", "kind", "dedupeKey", "attempts", "lastError", "createdAt"
FROM "BackgroundJob"
WHERE "queue" = 'notifications'
  AND "kind" = 'push_delivery'
ORDER BY "createdAt" DESC
LIMIT 50;
```

Delivery outcomes for a notification:

```sql
SELECT "status", COUNT(*) AS count
FROM "PushDelivery"
WHERE "notificationId" = $1
GROUP BY "status"
ORDER BY "status";
```

Duplicate successful sends to audit before adding a uniqueness constraint:

```sql
SELECT "notificationId", "deviceId", COUNT(*) AS sent_count
FROM "PushDelivery"
WHERE "status" = 'SENT'
GROUP BY "notificationId", "deviceId"
HAVING COUNT(*) > 1
ORDER BY sent_count DESC
LIMIT 50;
```

Invalidated device audit:

```sql
SELECT "status", COUNT(*) AS count
FROM "PushDevice"
GROUP BY "status"
ORDER BY "status";
```

## Rollback

If queued push delivery fails in staging:

1. Set `PUSH_DELIVERY_QUEUE_ENABLED=false`.
2. Stop the notifications worker.
3. Keep queued jobs and `PushDelivery` rows for inspection.
4. Re-enable inline `deliverPushForNotification()` until job processing is fixed.

If production rollout fails after jobs are enqueued, do not delete queued jobs until operators decide
whether they should be retried or intentionally abandoned.

## Production checklist

- A3.1 queue migration and worker are deployed
- push worker can be run independently from web request handling
- feature flag is available and defaults off
- staging verifies route responses are not blocked by provider latency
- retry/dead-letter behavior is verified
- duplicate-send audit is clean or explicitly accepted
- provider failure logs do not leak tokens or PII
- web gates pass: `pnpm --filter @zook/web typecheck`, `lint`, and `test`
- real-device push delivery remains listed for human/device QA

## Out of scope in this proposal

- implementing the durable queue table (`A3.1`)
- moving platform broadcast fan-out (`A3.2`)
- adding push-provider-specific rate-limit tuning
- certifying physical push notification delivery on real devices
