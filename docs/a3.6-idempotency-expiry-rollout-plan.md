# A3.6 Idempotency Expiry Rollout Plan

Last updated: 18 June 2026

## Current state

`RequestIdempotency` already exists and has the replay key required by the API runtime:

- `@@unique([userId, operation, requestHash])`
- `@@index([createdAt])`
- persisted response `status` and `body`

The runtime now uses the generated Prisma delegate directly and covers refund, cancel, and switch routes behind the `Idempotency-Key` header. The remaining schema work is adding explicit expiry metadata and a purge job.

## Proposed schema change

Add an explicit expiry timestamp to `RequestIdempotency`:

```prisma
model RequestIdempotency {
  id          String   @id @default(cuid())
  userId      String?
  operation   String
  requestHash String
  resultHash  String
  status      Int
  body        Json
  expiresAt   DateTime
  createdAt   DateTime @default(now())

  @@unique([userId, operation, requestHash])
  @@index([createdAt])
  @@index([expiresAt])
}
```

```sql
BEGIN;

ALTER TABLE "RequestIdempotency"
  ADD COLUMN IF NOT EXISTS "expiresAt" TIMESTAMP(3);

UPDATE "RequestIdempotency"
SET "expiresAt" = "createdAt" + INTERVAL '24 hours'
WHERE "expiresAt" IS NULL;

ALTER TABLE "RequestIdempotency"
  ALTER COLUMN "expiresAt" SET NOT NULL;

CREATE INDEX IF NOT EXISTS "RequestIdempotency_expiresAt_idx"
  ON "RequestIdempotency"("expiresAt");

COMMIT;
```

## Purge job

After the migration is verified, add a cron/admin job that deletes expired rows in bounded batches:

```sql
DELETE FROM "RequestIdempotency"
WHERE "expiresAt" < NOW()
  AND "id" IN (
    SELECT "id"
    FROM "RequestIdempotency"
    WHERE "expiresAt" < NOW()
    ORDER BY "expiresAt" ASC
    LIMIT 1000
  );
```

## Staging validation checklist

- Count current rows:
  `SELECT COUNT(*) FROM "RequestIdempotency";`
- Confirm backfill age window:
  `SELECT MIN("expiresAt"), MAX("expiresAt") FROM "RequestIdempotency";`
- Confirm no nulls remain:
  `SELECT COUNT(*) FROM "RequestIdempotency" WHERE "expiresAt" IS NULL;`
- Confirm the replay path still returns `x-idempotency-replay: true` for duplicate requests.
- Confirm purge deletes only rows older than the intended 24-hour window.

## Production rollout guardrails

- Run staging first with a snapshot or point-in-time restore available.
- Keep the existing `createdAt` predicate in runtime until the `expiresAt` column and purge job have been deployed everywhere.
- Deploy runtime support before switching the lookup predicate from `createdAt >= now - 24h` to `expiresAt > now`.
- Only then move the runtime to write `expiresAt` and read by `expiresAt`.
