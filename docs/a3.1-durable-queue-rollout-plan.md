# A3.1 Durable Queue Rollout Plan

Last updated: 18 June 2026

## Goal

Stand up a durable background job queue so request handlers stop doing slow fan-out and provider
work inline.

This is Product Plan Part A item `A3.1`, and it unblocks:

- `A3.2` platform broadcast fan-out off the request thread
- `A3.3` async push delivery
- the remaining queue portion of `A3.4` renewal reminders

This document is intentionally a rollout proposal only. It does not add a dependency, create a queue
table, or change runtime behavior because the queue backend has operational consequences for deploys,
cron, retries, and local development.

## Current evidence

There is no durable queue dependency or generic job table today:

- no `pg-boss`, `bullmq`, or `inngest` dependency in `package.json`
- queue-like models exist only for specific privacy jobs (`DataExportJob`, `AccountDeletionJob`)
- `fanOutPlatformBroadcast()` sleeps for 60 seconds between chunks inside the request path
- `createDirectNotification()` immediately calls `deliverPushForNotification()`
- cron jobs perform work inline inside `apps/web/src/server/api-router/core.ts`

The main problematic inline paths are:

- `apps/web/src/server/api-router/core.ts` (`fanOutPlatformBroadcast`)
- `apps/web/src/server/api-router/core.ts` (`createDirectNotification`)
- `apps/web/src/server/push-runtime.ts` (`deliverPushForNotification`)
- `apps/web/src/server/api-router/core.ts` (`cron/renewal-reminders`)

## Recommended backend

Prefer a Postgres-backed queue for the first production pass.

Rationale:

- Zook already depends on Postgres and Prisma.
- Queue data stays in the same backup/restore and staging-clone flow as application data.
- This avoids adding Redis/BullMQ operational requirements before the app has dedicated workers.
- It works for serverless-friendly cron polling as well as long-running workers later.

Two viable options:

1. Use `pg-boss`.
2. Build a small first-party `BackgroundJob` table with `FOR UPDATE SKIP LOCKED`.

Recommendation: start with a first-party table because it can be represented in Prisma, is easy to
dry-run, and keeps the first queue migration auditable. If throughput grows, move the worker
implementation to `pg-boss` while preserving the same job envelope fields.

## Proposed schema

Initial Prisma model:

```prisma
enum BackgroundJobStatus {
  QUEUED
  RUNNING
  SUCCEEDED
  FAILED
  DEAD
  CANCELLED
}

model BackgroundJob {
  id             String              @id @default(cuid())
  orgId          String?
  queue          String
  kind           String
  status         BackgroundJobStatus @default(QUEUED)
  dedupeKey      String?
  payload        Json
  result         Json?
  attempts       Int                 @default(0)
  maxAttempts    Int                 @default(5)
  runAt          DateTime            @default(now())
  lockedAt       DateTime?
  lockedBy       String?
  lastError      String?
  completedAt    DateTime?
  expiresAt      DateTime?
  createdById    String?
  createdAt      DateTime            @default(now())
  updatedAt      DateTime            @updatedAt

  @@unique([queue, dedupeKey])
  @@index([status, queue, runAt])
  @@index([orgId, status, createdAt])
  @@index([expiresAt])
}
```

If `dedupeKey` must remain nullable, use raw SQL for partial uniqueness instead of Prisma's nullable
compound unique:

```sql
CREATE UNIQUE INDEX background_job_queue_dedupe_unique_idx
ON "BackgroundJob" ("queue", "dedupeKey")
WHERE "dedupeKey" IS NOT NULL;
```

## Staging dry-run SQL

Do not apply directly to production. Run on staging first:

```sql
CREATE TYPE "BackgroundJobStatus" AS ENUM (
  'QUEUED',
  'RUNNING',
  'SUCCEEDED',
  'FAILED',
  'DEAD',
  'CANCELLED'
);

CREATE TABLE "BackgroundJob" (
  "id" text PRIMARY KEY,
  "orgId" text,
  "queue" text NOT NULL,
  "kind" text NOT NULL,
  "status" "BackgroundJobStatus" NOT NULL DEFAULT 'QUEUED',
  "dedupeKey" text,
  "payload" jsonb NOT NULL,
  "result" jsonb,
  "attempts" integer NOT NULL DEFAULT 0,
  "maxAttempts" integer NOT NULL DEFAULT 5,
  "runAt" timestamp(3) NOT NULL DEFAULT now(),
  "lockedAt" timestamp(3),
  "lockedBy" text,
  "lastError" text,
  "completedAt" timestamp(3),
  "expiresAt" timestamp(3),
  "createdById" text,
  "createdAt" timestamp(3) NOT NULL DEFAULT now(),
  "updatedAt" timestamp(3) NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX background_job_queue_dedupe_unique_idx
ON "BackgroundJob" ("queue", "dedupeKey")
WHERE "dedupeKey" IS NOT NULL;

CREATE INDEX "BackgroundJob_status_queue_runAt_idx"
ON "BackgroundJob" ("status", "queue", "runAt");

CREATE INDEX "BackgroundJob_orgId_status_createdAt_idx"
ON "BackgroundJob" ("orgId", "status", "createdAt");

CREATE INDEX "BackgroundJob_expiresAt_idx"
ON "BackgroundJob" ("expiresAt");
```

Validation:

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'BackgroundJob'
ORDER BY indexname;
```

## Worker claim query

Workers should claim bounded batches atomically:

```sql
WITH claim AS (
  SELECT "id"
  FROM "BackgroundJob"
  WHERE "status" = 'QUEUED'
    AND "queue" = $1
    AND "runAt" <= NOW()
  ORDER BY "runAt" ASC, "createdAt" ASC
  LIMIT $2
  FOR UPDATE SKIP LOCKED
)
UPDATE "BackgroundJob" job
SET
  "status" = 'RUNNING',
  "lockedAt" = NOW(),
  "lockedBy" = $3,
  "attempts" = job."attempts" + 1,
  "updatedAt" = NOW()
FROM claim
WHERE job."id" = claim."id"
RETURNING job.*;
```

Failed jobs should be rescheduled with exponential backoff until `attempts >= maxAttempts`, then
marked `DEAD`.

## Queue kinds

Start with these queue names and job kinds:

| Queue | Kind | Producer | Consumer |
| --- | --- | --- | --- |
| `notifications` | `platform_broadcast_fanout` | platform broadcast create/update | fan out in recipient chunks |
| `notifications` | `notification_push_delivery` | notification create/resend | Expo/mock push provider delivery |
| `reminders` | `renewal_reminder_batch` | renewal cron | create reminders and notifications |
| `maintenance` | `retention_purge` | retention cron | purge expired operational rows |

Keep payloads small. Store large generated files or external payloads in domain tables and pass IDs in
the job payload.

## Runtime API shape

Add server helpers before moving call sites:

```ts
enqueueBackgroundJob({
  queue: "notifications",
  kind: "platform_broadcast_fanout",
  dedupeKey: `platform_broadcast:${broadcast.id}`,
  orgId,
  payload,
  runAt,
});

claimBackgroundJobs({ queue: "notifications", workerId, limit: 25 });

completeBackgroundJob(jobId, result);

failBackgroundJob(jobId, error);
```

The enqueue helper must:

- validate payload shape with Zod per `kind`
- set `expiresAt` according to queue retention policy
- upsert or ignore duplicates by `dedupeKey`
- return the existing queued job for idempotent replays

## Rollout phases

### Phase 1: table and helpers

- add `BackgroundJob` schema/migration after staging dry-run
- add enqueue/claim/complete/fail helpers
- add unit tests for dedupe, backoff, and dead-letter transitions
- add a protected worker endpoint or script that can process a single queue

### Phase 2: platform broadcasts (`A3.2`)

- replace inline `fanOutPlatformBroadcast()` request work with enqueue
- return `{ broadcast, fanoutJobId }` from API responses
- worker chunks recipients and respects provider rate limits without request sleeps
- keep audit logs on enqueue and completion

### Phase 3: push delivery (`A3.3`)

- replace immediate `deliverPushForNotification()` calls with push jobs
- worker records `PushDelivery` rows
- provider failures retry with backoff
- invalid device tokens are still revoked

### Phase 4: renewal reminders

- cron enqueues reminder batches
- worker performs reminder creation and notification enqueue
- cron response becomes fast and idempotent

## Operational requirements

- `CRON_SECRET` protects worker/cron endpoints.
- At most one worker batch should claim the same job due to `FOR UPDATE SKIP LOCKED`.
- Worker identity should include deployment/version/pid for debugging.
- Dead-letter jobs must be visible in platform operations.
- Queue depth, oldest queued age, failure rate, and dead count should be logged or exposed in
  provider/platform diagnostics.

## Validation checklist

Before production rollout:

- staging migration creates table/indexes successfully
- enqueue dedupe prevents duplicate jobs for the same broadcast
- two concurrent workers cannot claim the same job
- failed jobs retry and then dead-letter after `maxAttempts`
- platform broadcast request returns quickly without 60-second sleeps
- push delivery failures are recorded and retried
- renewal cron no longer performs full fan-out inline
- web gates pass: `pnpm --filter @zook/web typecheck`, `lint`, and `test`
- root `pnpm typecheck` passes if Prisma/generated DB types change

## Rollback

Before producers depend on the queue:

```sql
DROP TABLE IF EXISTS "BackgroundJob";
DROP TYPE IF EXISTS "BackgroundJobStatus";
```

After producers enqueue jobs, rollback must first deploy code that falls back to inline execution or
temporarily pauses producers. Export unfinished jobs before dropping the table:

```sql
SELECT *
FROM "BackgroundJob"
WHERE "status" IN ('QUEUED', 'RUNNING', 'FAILED')
ORDER BY "createdAt" ASC;
```

## Out of scope in this proposal

- moving platform broadcast fan-out (`A3.2`)
- moving push delivery (`A3.3`)
- retention purge implementation from `B6`
- choosing a third-party queue provider such as BullMQ/Inngest
