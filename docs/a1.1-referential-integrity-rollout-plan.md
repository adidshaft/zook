# A1.1 Referential Integrity Rollout Plan

Last updated: 16 June 2026

## Goal

Implement the Part A P0 item `A1.1` from `zook-product-improvement-plan.md`:

- add database-level referential integrity across the schema
- add explicit `@relation` metadata and `onDelete` behavior
- stop silent orphan creation and unsafe deletes

The product plan’s intended policy is:

- `Cascade` for child-of-tenant records
- `Restrict` for financial records
- `SetNull` for nullable actor references

## Why this must be phased

The current schema is relation-light by design:

- effectively no Prisma `@relation` declarations
- no explicit `onDelete` behavior
- many foreign-key-like `...Id` columns across more than 100 models

That means this cannot be treated as one migration. We need:

1. relation inventory
2. orphan audits by category
3. staged rollout on a staging clone
4. category-by-category migrations with rollback points

## Rollout strategy

Split the work into four buckets instead of one schema-wide blast radius:

1. **Tenant tree**
   - org-owned and branch-owned children
   - examples: `Branch.orgId`, `OrganizationUser.orgId`, `AttendanceRecord.branchId`
   - default policy: `Cascade` from tenant roots unless business data needs stricter retention

2. **Operational domain records**
   - memberships, attendance, plans, assignments, notifications, QR tokens
   - mixed policies depending on whether rows are historical evidence or disposable children

3. **Financial records**
   - payments, refunds, invoices, mandates, payout ledgers
   - default policy: `Restrict`
   - do not allow deletes that would sever accounting history

4. **Nullable actor references**
   - audit-ish references to users who acted on a record
   - examples: `approvedById`, `rejectedById`, `recordedById`, `assignedById`
   - default policy: `SetNull`

Do not try to land all four buckets at once.

## Immediate schema evidence

Examples of current FK-like fields without declared relations:

- tenant roots / branch scope
  - `Branch.orgId`
  - `OrganizationUser.orgId`
  - `OrganizationRoleAssignment.orgId`
  - `AttendanceRecord.orgId`, `AttendanceRecord.branchId`
  - `MemberSubscription.orgId`, `MemberSubscription.branchId`

- financial records

  - `PaymentRefund.paymentId`
  - `Invoice.paymentId`
  - `SubscriptionReminder.paymentId`, `paymentEventId`

- nullable actor references

This plan intentionally does not attempt to enumerate every FK candidate inline; the first staging pass should generate a reviewed inventory.

## Phase 0: inventory pass

Before writing migrations, build a table of candidate FK relationships:

- table name
- column name
- target table
- nullable vs non-nullable
- proposed `onDelete`
- whether existing orphan cleanup is required
- rollout bucket

Suggested local inventory helper:

```bash
rg -n 'Id\\s+String\\??' packages/db/prisma/schema.prisma
```

Then hand-curate the results into a checklist before any migration is written.

## Phase 1: orphan audits

Run orphan audits on a staging clone first. Here are representative queries to use as templates.

### Tenant / branch examples

Branches pointing to missing orgs:

```sql
SELECT b."id", b."orgId", b."name"
FROM "Branch" b
LEFT JOIN "Organization" o ON o."id" = b."orgId"
WHERE o."id" IS NULL;
```

Organization users pointing to missing users:

```sql
SELECT ou."id", ou."orgId", ou."userId"
FROM "OrganizationUser" ou
LEFT JOIN "User" u ON u."id" = ou."userId"
WHERE u."id" IS NULL;
```

Attendance rows pointing to missing branches:

```sql
SELECT ar."id", ar."orgId", ar."branchId", ar."userId", ar."dateKey"
FROM "AttendanceRecord" ar
LEFT JOIN "Branch" b ON b."id" = ar."branchId"
WHERE b."id" IS NULL;
```

### Financial examples

Refunds pointing to missing payments:

```sql
SELECT pr."id", pr."paymentId", pr."orgId", pr."amountPaise", pr."status"
FROM "PaymentRefund" pr
LEFT JOIN "Payment" p ON p."id" = pr."paymentId"
WHERE p."id" IS NULL;
```

Invoices pointing to missing payments:

```sql
SELECT i."id", i."paymentId", i."orgId", i."number"
FROM "Invoice" i
LEFT JOIN "Payment" p ON p."id" = i."paymentId"
WHERE i."paymentId" IS NOT NULL
  AND p."id" IS NULL;
```

### Nullable actor examples

Attendance approvers that no longer exist:

```sql
SELECT ar."id", ar."approvedById"
FROM "AttendanceRecord" ar
LEFT JOIN "User" u ON u."id" = ar."approvedById"
WHERE ar."approvedById" IS NOT NULL
  AND u."id" IS NULL;
```

Payments recorded by missing users:

```sql
SELECT p."id", p."recordedById", p."orgId", p."amountPaise"
FROM "Payment" p
LEFT JOIN "User" u ON u."id" = p."recordedById"
WHERE p."recordedById" IS NOT NULL
  AND u."id" IS NULL;
```

Do not proceed on any bucket until its orphan audits are understood and documented.

## Proposed delete-policy defaults

These are defaults, not a blind rule:

### `Cascade`

Use for rows that are clearly children of a tenant/container and should not outlive it.

Examples to evaluate:

- `Branch.orgId -> Organization.id`
- `OrganizationUser.orgId -> Organization.id`
- `OrganizationRolePermission.orgId -> Organization.id`
- `AttendanceQrToken.branchId -> Branch.id`
- `ClassEnrollment.classId -> Class.id`
- `ShopOrderItem.orderId -> ShopOrder.id`

### `Restrict`

Use for financial and compliance history.

Examples to evaluate:

- `PaymentRefund.paymentId -> Payment.id`
- `Invoice.paymentId -> Payment.id`
- `PaymentEvent.paymentId -> Payment.id`
- `TrainerPayoutLine.payoutId -> TrainerPayout.id`
- `ManualPaymentAdjustment.originalPaymentId -> Payment.id`

If a parent row cannot safely disappear without rewriting history, prefer `Restrict`.

### `SetNull`

Use for optional actor references where history should survive user deletion/deactivation.

Examples to evaluate:

- `AttendanceRecord.approvedById -> User.id`
- `AttendanceRecord.rejectedById -> User.id`
- `Payment.recordedById -> User.id`
- `PaymentRefund.requestedById -> User.id`
- `MembershipJoinRequest.reviewedById -> User.id`
- `StaffInvitation.acceptedById -> User.id`

## Staging dry-run procedure

1. Clone staging from the latest production snapshot.
2. Build the FK inventory sheet for the selected rollout bucket.
3. Run orphan audits for that bucket.
4. If any orphan rows exist, document and clean them on staging only.
5. Add the Prisma `@relation` metadata and/or manual SQL for that bucket.
6. Generate the migration.
7. Apply it on staging only.
8. Re-run core smoke tests for that bucket:
   - create/read/update flows still work
   - deletes fail or cascade exactly as intended
   - no provider/webhook/financial history is severed accidentally

Do this bucket-by-bucket, not globally.

## Migration authoring notes

For each bucket:

- prefer small, named migrations
- include a human-readable migration note describing why the chosen `onDelete` is safe
- where Prisma cannot express the exact intent cleanly, use manual SQL in the migration

Do not mix:

- tenant cascades
- financial restrict constraints
- actor `SetNull`

in the same first-pass rollout unless staging evidence proves the batch is safe.

## Rollback

Rollback should be planned per bucket.

At minimum:

- restore from snapshot if a constraint batch causes unexpected application failures
- drop the newly added foreign keys for the affected bucket
- mark the migration rolled back only after the schema state is confirmed

Because FK names differ per migration tool output, capture the exact generated constraint names in the staging dry-run notes before production.

## Production rollout checklist

- `pnpm release:preflight` passes with the target env
- fresh database backup/snapshot exists
- the exact FK bucket being deployed has a reviewed inventory sheet
- orphan audits returned zero unresolved rows on the rollout target
- staging dry-run evidence is attached
- rollback names/commands are recorded before production apply
- financial buckets are deployed separately from tenant cascades

## Recommended sequencing after this proposal

1. Tenant/branch tree with low historical risk
2. Nullable actor `SetNull` references
3. Operational domain relations
4. Financial `Restrict` relations last, in narrow batches

## Out of scope in this proposal

- direct application of any FK migration
- `A1.8` manual payment amount validation
- Part B enum/schema cleanup

This document is the rollout plan, not the migration itself.
