# A3.2 Broadcast Fan-Out Queue Rollout Plan

Last updated: 18 June 2026

## Goal

Move platform broadcast fan-out off the HTTP request thread.

This is Product Plan Part A item `A3.2`.

This document is intentionally a rollout proposal only. It does not change runtime behavior because
the required durable queue from `A3.1` has not been migrated/applied yet, and changing the platform
broadcast response from immediate `fanout` counts to queued job status is an API contract change that
should be coordinated with the platform console.

## Current problem

`fanOutPlatformBroadcast()` still runs in the platform broadcast create/update request path:

```ts
if (chunksSent > 0) {
  await sleep(60_000);
}
const notification = await createDirectNotification({ ... });
```

This creates several production risks:

- a broadcast to more than one 500-user chunk waits at least 60 seconds inside the request
- retries can duplicate fan-out work because the request may time out after partially sending
- push delivery is coupled to the platform admin request path through `createDirectNotification()`
- platform admins do not get a durable job state they can inspect or retry

## Dependency on A3.1

`A3.2` should use the durable queue proposed in:

- `docs/a3.1-durable-queue-rollout-plan.md`

Required queue kind:

```ts
{
  queue: "notifications",
  kind: "platform_broadcast_fanout",
  dedupeKey: `platform_broadcast:${broadcast.id}`,
}
```

Do not remove inline fan-out until the queue table, worker claim path, and worker endpoint/script are
available in staging.

## Desired API behavior

Current API response shape:

```ts
return ok({ broadcast, fanout });
```

Target response after the queue rollout:

```ts
return ok({
  broadcast,
  fanout: null,
  fanoutJob: {
    id: job.id,
    status: job.status,
    queued: true,
  },
});
```

This is additive if `fanout` remains present as `null`, but the platform console must be updated to
show queued/in-progress/completed states instead of assuming counts are immediately final.

## Job payload

Use a small payload and re-read the broadcast when the worker runs:

```ts
{
  broadcastId: string;
  createdByUserId: string;
  requestedAt: string;
}
```

The worker should load `PlatformBroadcast` by `broadcastId` and validate:

- broadcast still exists
- status is `LIVE`
- `publishedAt` is set or can be set idempotently
- `targetOrgIds`, `targetRoles`, `title`, `body`, and `severity` are read from the current row

## Producer changes

In `POST /api/platform/broadcasts`:

1. Create the broadcast.
2. If status is not `LIVE`, return with no job.
3. If status is `LIVE`, enqueue `platform_broadcast_fanout`.
4. Write audit action `platform.broadcast_created` with `{ fanoutJobId }`.
5. Return immediately.

In `PATCH /api/platform/broadcasts/:id`:

1. Update the broadcast.
2. Only enqueue when transitioning from non-`LIVE` to `LIVE`.
3. Use the same dedupe key so duplicate publish attempts do not enqueue duplicate fan-out jobs.
4. Write audit action `platform.broadcast_updated` with `{ fanoutJobId }`.

## Worker behavior

The worker should:

1. Resolve recipients with the existing `resolvePlatformBroadcastRecipients()`.
2. Chunk by 500 users.
3. Create one notification per org/chunk through `createDirectNotification()` or a lower-level helper.
4. Wait between chunks in the worker, not in the HTTP request.
5. Record progress in the job result after each chunk.
6. Retry failed chunks with exponential backoff.
7. Mark the job `DEAD` after `maxAttempts`.

Recommended job result shape:

```ts
{
  broadcastId: string;
  notifications: number;
  recipients: number;
  chunks: number;
  completedChunks: number;
  failedChunks: Array<{
    orgId: string;
    chunkIndex: number;
    error: string;
  }>;
}
```

## Idempotency rules

Fan-out must be safe to retry.

Recommended dedupe markers:

- job dedupe: `platform_broadcast:${broadcast.id}`
- notification metadata:
  - `platformBroadcastId`
  - `platformBroadcastChunkKey`
  - `platformBroadcastChunkIndex`

Before creating a notification for a chunk, the worker should check whether a notification already
exists with matching `platformBroadcastId` and `platformBroadcastChunkKey` in metadata. If yes, skip
that chunk and count it as already complete.

## Platform console updates

Before enabling queued fan-out, update the platform broadcast UI to show:

- queued job id
- status (`QUEUED`, `RUNNING`, `SUCCEEDED`, `FAILED`, `DEAD`)
- recipient count so far
- completed chunks
- last error
- retry/dead-letter guidance for platform admins

This can be backed by a read-only `GET /api/platform/jobs?queue=notifications&kind=platform_broadcast_fanout`
endpoint after A3.1 lands.

## Staging rollout steps

1. Apply and verify the A3.1 `BackgroundJob` migration on staging.
2. Deploy queue helpers and a worker endpoint/script.
3. Add producer code behind a feature flag:
   - `PLATFORM_BROADCAST_QUEUE_ENABLED=false` by default
4. Run with the flag disabled and confirm current inline behavior is unchanged.
5. Enable the flag on staging.
6. Create a draft broadcast and publish it.
7. Confirm the HTTP response returns immediately with a job id.
8. Run the worker and verify notifications/recipients are created exactly once.
9. Replay the same publish request or patch and confirm no duplicate job/chunks.
10. Force a worker failure and confirm retry/dead-letter behavior.

## Validation queries

Queued job:

```sql
SELECT "id", "status", "queue", "kind", "dedupeKey", "attempts", "result", "lastError"
FROM "BackgroundJob"
WHERE "queue" = 'notifications'
  AND "kind" = 'platform_broadcast_fanout'
ORDER BY "createdAt" DESC
LIMIT 20;
```

Chunk notifications:

```sql
SELECT
  "orgId",
  COUNT(*) AS notification_count,
  SUM(jsonb_array_length(COALESCE("metadata"->'selectedUserIds', '[]'::jsonb))) AS selected_users
FROM "Notification"
WHERE "metadata"->>'platformBroadcastId' = $1
GROUP BY "orgId"
ORDER BY "orgId";
```

Duplicate chunk audit:

```sql
SELECT
  "metadata"->>'platformBroadcastChunkKey' AS chunk_key,
  COUNT(*) AS duplicate_count
FROM "Notification"
WHERE "metadata"->>'platformBroadcastId' = $1
GROUP BY chunk_key
HAVING COUNT(*) > 1;
```

## Rollback

If queued fan-out fails in staging:

1. Set `PLATFORM_BROADCAST_QUEUE_ENABLED=false`.
2. Stop the worker.
3. Leave queued jobs in place for inspection.
4. Re-enable inline fan-out until job processing is fixed.

If production rollout fails after jobs are enqueued, do not delete jobs until platform admins confirm
whether partial fan-out has already happened.

## Production checklist

- A3.1 queue migration and worker are deployed
- platform console can show queued fan-out state
- feature flag is available and defaults off
- staging publish response returns without request-thread delay
- duplicate publish attempts do not duplicate notification chunks
- worker retry/dead-letter behavior is verified
- web gates pass: `pnpm --filter @zook/web typecheck`, `lint`, and `test`
- product/platform admin copy is updated for queued vs completed broadcast state

## Out of scope in this proposal

- implementing the durable queue table (`A3.1`)
- moving push delivery to jobs (`A3.3`)
- changing provider push rate limits
- building full platform job-management UI
