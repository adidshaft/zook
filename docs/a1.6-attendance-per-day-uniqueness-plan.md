# A1.6 Attendance Per-Day Uniqueness Plan

Last updated: 18 June 2026

## Goal

Enforce the product-plan invariant for attendance:

- one attendance record per `(orgId, branchId, userId, dateKey)` day scope

This is the Part A P0 item `A1.6` from `zook-product-improvement-plan.md`.

## Why this is still needed

Current behavior already tries to block duplicate same-day check-ins in application code:

- QR scan checks for `todayCheckIn`
- `validateAttendanceScan()` rejects `already_checked_in`
- visit consumption also checks whether the member already checked in today

But the database does not enforce this invariant. The current schema only has a non-unique index:

```prisma
@@index([orgId, branchId, userId, dateKey])
```

That means concurrent writes, retries, or future route regressions can still create same-day duplicate rows.

## Current schema target

Current model excerpt from `packages/db/prisma/schema.prisma`:

```prisma
model AttendanceRecord {
  id             String           @id @default(cuid())
  orgId          String
  branchId       String
  userId         String
  subscriptionId String?
  status         AttendanceStatus @default(PENDING_APPROVAL)
  dateKey        String
  ...

  @@index([orgId, branchId, userId, dateKey])
  ...
}
```

Desired invariant from the plan:

```sql
CREATE UNIQUE INDEX attendance_record_one_per_day_idx
ON "AttendanceRecord" ("orgId", "branchId", "userId", "dateKey");
```

## Product caveat

This is only safe if the product intent is truly "one record per member per branch per day", regardless of status.

Today the code path suggests that is the intended behavior:

- a second same-day check-in is treated as a duplicate, not a new visit
- `consumeVisit()` defaults to not consuming another visit for same-day re-entry

But before production rollout, verify there is no intended workflow that requires multiple attendance rows for the same person/branch/day, such as:

- rejected attempt plus approved retry on the same day
- separate manual-override row instead of updating the original row
- multi-session gyms that want two distinct records in one day

If product wants any of those, the index should likely be partial or status-scoped instead of global.

## Preflight duplicate audit

Generate the full preflight/staging SQL pack with:

```bash
pnpm db:a1.6:sql
```

You can print only one section with:

```bash
pnpm db:a1.6:sql duplicate-audit
pnpm db:a1.6:sql detail-query
pnpm db:a1.6:sql create-index-staging-only
pnpm db:a1.6:sql validate-index
pnpm db:a1.6:sql rollback
```

Run the duplicate audit first on a staging clone or disposable copy of production:

```sql
SELECT
  "orgId",
  "branchId",
  "userId",
  "dateKey",
  COUNT(*) AS record_count,
  ARRAY_AGG("id" ORDER BY "checkedInAt" ASC, "createdAt" ASC) AS attendance_ids
FROM "AttendanceRecord"
GROUP BY "orgId", "branchId", "userId", "dateKey"
HAVING COUNT(*) > 1
ORDER BY record_count DESC, "orgId", "branchId", "userId", "dateKey";
```

If this returns any rows, do not apply the index yet.

Detail query per collision set:

```sql
SELECT
  "id",
  "orgId",
  "branchId",
  "userId",
  "subscriptionId",
  "status",
  "source",
  "dateKey",
  "checkedInAt",
  "checkedOutAt",
  "approvedAt",
  "approvedById",
  "rejectedAt",
  "rejectedById",
  "rejectionReason",
  "suspiciousFlags",
  "qrTokenId",
  "deviceId",
  "createdAt"
FROM "AttendanceRecord"
WHERE "orgId" = $1
  AND "branchId" = $2
  AND "userId" = $3
  AND "dateKey" = $4
ORDER BY "checkedInAt" ASC, "createdAt" ASC;
```

## Suggested remediation rule on staging

Do not auto-collapse duplicates blindly.

Review each set using this order:

1. Keep the row that best represents the real visit outcome:
   usually the approved row with the most complete timestamps.
2. If there is one approved row and one pending/rejected duplicate for the same visit window, preserve the approved row.
3. Prefer updating or archiving extra rows after review rather than deleting them without an audit trail.

Because attendance records are operational/audit data, cleanup should be reviewed with product/ops if duplicates are found.

## Staging dry-run procedure

1. Restore or clone staging from the latest production snapshot.
2. Run the duplicate audit query above.
3. If duplicates exist, document the cleanup rule used for each group.
4. Apply cleanup on staging only.
5. Re-run the duplicate audit until it returns zero rows.
6. Apply the index SQL on staging.
7. Re-run attendance smoke tests:
   - first QR check-in succeeds
   - second same-day QR check-in is rejected or treated as the existing visit
   - checkout still works on the kept row
   - manual attendance override still works as intended
   - same member can still check in on a different `dateKey`

Index application SQL for staging:

```sql
CREATE UNIQUE INDEX attendance_record_one_per_day_idx
ON "AttendanceRecord" ("orgId", "branchId", "userId", "dateKey");
```

Validation query after index creation:

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'AttendanceRecord'
  AND indexname = 'attendance_record_one_per_day_idx';
```

## Prisma migration note

This one can be represented as a normal Prisma uniqueness change, but it is still a production-sensitive constraint because it can fail on existing duplicate rows.

Treat it as:

1. a named migration after duplicate cleanup is proven on staging, and
2. a production rollout only after a fresh duplicate audit on the target database

Do not apply it directly to production without the staging dry-run and duplicate review.

## Rollback

If the index needs to be reversed:

```sql
DROP INDEX IF EXISTS attendance_record_one_per_day_idx;
```

If recorded through Prisma, follow the repo runbook and resolve the named migration as rolled back only after the index is removed and the database state is verified.

## Production rollout checklist

- `pnpm release:preflight` passes with the target env
- fresh database backup/snapshot exists
- staging clone dry-run is documented
- duplicate audit returned zero rows on the rollout target immediately before applying the index
- QR/manual attendance smoke checks were re-run after staging index creation
- ops/product explicitly confirmed that one row per member/branch/day is the desired invariant
