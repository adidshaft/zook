# A1.5 Single Active Subscription Migration Plan

Last updated: 16 June 2026

## Goal

Enforce the product-plan invariant for memberships:

- at most one `ACTIVE` `MemberSubscription` per `(orgId, branchId, memberUserId)`

This is the Part A P0 item `A1.5` from `zook-product-improvement-plan.md`.

## Why this needs a staged rollout

The current schema has:

- no partial unique index on `MemberSubscription`
- no DB-level guard against concurrent activation races

The application now has better replay safety in payment activation, but it still depends on app logic alone. Adding the index is the durable fix.

Because this is a constraint on existing data, we should not auto-apply it locally or directly against production. We need:

1. duplicate preflight queries
2. remediation on a staging clone if duplicates exist
3. a staging dry-run of the exact SQL
4. only then a production rollout window

## Current schema target

Current model excerpt from `packages/db/prisma/schema.prisma`:

```prisma
model MemberSubscription {
  id              String             @id @default(cuid())
  orgId           String
  branchId        String
  memberUserId    String
  planId          String
  status          SubscriptionStatus @default(PENDING_PAYMENT)
  ...

  @@index([orgId, memberUserId, status])
  @@index([orgId, branchId, memberUserId])
  @@index([orgId, status])
  @@index([orgId, planId])
  @@index([endsAt])
}
```

Desired invariant:

```sql
CREATE UNIQUE INDEX member_subscription_single_active_idx
ON "MemberSubscription" ("orgId", "branchId", "memberUserId")
WHERE status = 'ACTIVE';
```

## Preflight duplicate audit

Run this first on a staging clone or disposable copy of production:

```sql
SELECT
  "orgId",
  "branchId",
  "memberUserId",
  COUNT(*) AS active_count,
  ARRAY_AGG("id" ORDER BY "createdAt" ASC) AS subscription_ids
FROM "MemberSubscription"
WHERE status = 'ACTIVE'
GROUP BY "orgId", "branchId", "memberUserId"
HAVING COUNT(*) > 1
ORDER BY active_count DESC, "orgId", "branchId", "memberUserId";
```

If this returns any rows, do not apply the index yet.

Useful detail query for each collision set:

```sql
SELECT
  "id",
  "orgId",
  "branchId",
  "memberUserId",
  "planId",
  "status",
  "startsAt",
  "endsAt",
  "paymentId",
  "activatedById",
  "createdAt",
  "updatedAt"
FROM "MemberSubscription"
WHERE "orgId" = $1
  AND "branchId" = $2
  AND "memberUserId" = $3
ORDER BY
  CASE WHEN status = 'ACTIVE' THEN 0 ELSE 1 END,
  "createdAt" ASC;
```

## Suggested remediation rule on staging

Do not apply a blanket cleanup automatically.

Use this review order for each duplicate set:

1. Keep the row tied to the successful/latest intended payment when that is unambiguous.
2. Prefer the row with the most credible activation metadata:
   `paymentId`, `activatedById`, valid `startsAt`/`endsAt`, and expected `remainingVisits`.
3. Mark the extra rows non-active rather than deleting them:
   typically `CANCELLED` or another historically correct terminal state after product review.

This needs human review because changing a subscription status is business logic, not just schema cleanup.

## Staging dry-run procedure

1. Restore or clone staging from the latest production snapshot.
2. Run the duplicate audit query above.
3. If duplicates exist, document each remediation decision before editing rows.
4. Apply the cleanup on staging only.
5. Re-run the duplicate audit until it returns zero rows.
6. Apply the index SQL on staging.
7. Re-run payment activation and renewal smoke tests:
   - successful new membership activation
   - successful renewal of a non-active/pending subscription
   - duplicate webhook replay acknowledgement
   - rejection of a second concurrent activation for the same member/branch

Index application SQL for staging:

```sql
CREATE UNIQUE INDEX member_subscription_single_active_idx
ON "MemberSubscription" ("orgId", "branchId", "memberUserId")
WHERE status = 'ACTIVE';
```

Validation query after index creation:

```sql
SELECT indexname, indexdef
FROM pg_indexes
WHERE tablename = 'MemberSubscription'
  AND indexname = 'member_subscription_single_active_idx';
```

## Prisma migration note

Prisma schema syntax does not express this partial unique index directly in the current model. The rollout should therefore be:

1. create a named migration with a manual SQL step, or
2. execute the SQL through the staged DB rollout process and then record the migration state carefully

Do not pretend this is a normal `@@unique` in `schema.prisma`; the partial `WHERE status = 'ACTIVE'` clause is the important part.

## Rollback

If staging or production rollout needs to be reversed:

```sql
DROP INDEX IF EXISTS member_subscription_single_active_idx;
```

If the migration is recorded through Prisma, follow the repo runbook guidance and resolve the named migration as rolled back only after the index is removed and the database state is confirmed.

## Production rollout checklist

- `pnpm release:preflight` passes with the target env
- fresh database backup/snapshot exists
- staging clone dry-run is documented
- duplicate audit returned zero rows on the rollout target immediately before applying the index
- support team knows activation collisions should now fail closed at the DB layer

## Out of scope in this proposal

- `A1.6` attendance per-day uniqueness
- `A1.7` visit-deduction uniqueness
- `A1.8` desk-payment amount validation

Those should be proposed separately so each rollout can have its own duplicate audit and remediation notes.
